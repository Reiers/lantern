## What Phase 11 shipped

### Part A — Bootstrap quorum (CRITICAL — the trust foundation)

Files:

- `chain/bootstrap/bootstrap.go` — `Source` interface, `Finality`,
  `Quorum()` driver, `QuorumResult`, `FormatReport()`. Pure-Go, no
  network deps; just orchestrates source goroutines and tallies
  equality buckets.
- `chain/bootstrap/bootstrap_test.go` — 9 test cases, all pass under
  `CGO_ENABLED=0`.
- `chain/bootstrap/sources/sources.go` — concrete `Source` impls:
  `ForestSource`, `LanternGatewaySource`, `UserPeerSource` (all
  Lotus-compatible JSON-RPC over HTTP), `Libp2pSource` (F3
  `/f3/certexch/get/1/<nn>`), `LanternBeaconSource` (stub returning
  `ErrNoBeaconBackend` until V1.2.1 ships cert-exchange-over-beacon).
- `chain/bootstrap/sources/discover.go` — `BuildDefaultSources()`
  helper + `MainnetPublicForestURLs` constant (glif + chain.love).
- `cmd/lantern/init.go` — `lantern init --bootstrap-quorum=N`
  wizard, `BootstrapAnchor` JSON file writer.
- `cmd/lantern/doctor.go` — `lantern doctor` (read-only quorum
  health check), `lantern repair` (re-anchor from fresh quorum).

The Lantern gateway is `KindLanternGateway` and **never counted by
default**. Operators who explicitly trust the project can opt in via
`--count-gateway`.

The `Source.LatestFinality()` interface decision was deliberate: every
source answers exactly one question — "what is the latest F3-finalized
tipset?" — so we can equality-compare answers across wildly different
transports (HTTP vs libp2p cert-exchange) without leaking transport
details into the quorum tally.

## Decisions worth review

- **Quorum equality uses the head tipset's `PowerTable` CID as
  `StateRoot`** in the `Finality` shape, not the chain state root.
  Light clients learn the chain state root by walking from the trust
  anchor; the bootstrap quorum's job is to agree on *which tipset is
  finalized*, not the state root. The `PowerTable` CID is included in
  every F3 cert's `ECChain.Head()` and serves as a deterministic
  agreement digest. Documented in `sources.go` `finalityFromCert()`.
- **`LanternBeaconSource` is a stub** that always returns
  `ErrNoBeaconBackend`. Beacons today serve Bitswap but do not yet
  implement cert-exchange or expose a JSON-RPC head endpoint. When
  V1.2.1 ships beacon cert-exchange, this becomes a real source and
  the quorum gains an additional independent class.
- **Default quorum is 5** (per INSTALLER-SPEC.md §3). Lower with
  `--bootstrap-quorum N`; the minimum sensible value is 3 (still
  beats every "single trusted RPC" deployment in the Filecoin
  ecosystem today). `--bootstrap-quorum 0` skips the check
  entirely; intended for testing only.
- **Two HTTP forest sources, not one.** The default set is glif +
  chain.love. Three libp2p peers + two HTTP archives = 5 healthy
  responders, which lands quorum=5 reliably under real mainnet
  network conditions (some libp2p bootstrap peers refuse
  cert-exchange or fail to dial).
- **The Lantern gateway always probes, never counts.** Including it
  as a non-counted source preserves the "every source we know about
  showed up in the report" UX without weakening the trust model.
  `--count-gateway` is the explicit opt-in.

## What Phase 11 did NOT ship (open items)

### B-11-01 — Beacon cert-exchange

- **Severity:** medium — affects `LanternBeaconSource` only.
- **What's missing:** Lantern beacons (the `lantern beacon`
  subcommand) need a cert-exchange responder so other clients can
  ask them for the latest F3 finality. Today the beacon serves
  Bitswap and announces under `lantern/beacon/v1` rendezvous, but
  doesn't speak F3 cert-exchange.
- **Plan:** wire `go-f3/certexchange.Server` into `cmd/lantern/beacon.go`,
  using the beacon's existing libp2p host. The data source is the
  beacon's own `chain/f3/subscriber.State()` (the certs it has
  already verified). Ship in V1.2.1.

### B-11-02 — Quorum status endpoint for Mac app

- **Severity:** low — UX polish.
- **What's missing:** The Mac app's quorum-status indicator (green /
  amber / red) is a recency heuristic on `bootstrap-anchor.json`,
  not a real-time quorum probe. A dedicated daemon endpoint
  (`Filecoin.NodeStatus` or `Lantern.QuorumStatus`) that exposes
  the current per-source agreement would let the indicator reflect
  live state.
- **Plan:** add a small RPC handler in `rpc/handlers/extra.go` that
  re-runs the quorum probe on demand (cached for ~60s). Mac app
  consumes that instead of the recency heuristic. V1.2.1.

### B-11-03 — Signed + notarised macOS release artifacts

- **Severity:** low — only blocks "drag-and-drop install" UX.
- **What's missing:** `Lantern.app` is built unsigned. Users have to
  `xattr -dr com.apple.quarantine /Applications/Lantern.app` to
  bypass Gatekeeper. The release workflow doesn't sign or notarise
  Mac binaries either.
- **Plan:** add Apple Developer ID + notarytool steps to the release
  workflow once we register an Apple Developer team for the
  project. V1.2.1 or V1.3 depending on priority.

### B-11-04 — Sparkle auto-update for Mac app

- **Severity:** low.
- **What's missing:** The Mac app doesn't auto-update. Users either
  re-run `install.sh` or rebuild from source for new versions.
- **Plan:** add Sparkle once we have signed release artifacts to
  point an appcast at. V1.3.

### B-11-05 — Settings pane in Mac app

- **Severity:** low.
- **What's missing:** The Mac app shows status but doesn't let the
  user edit quorum sources, JWT scopes, or toggle beacon mode.
- **Plan:** SwiftUI `Settings` scene next to `MenuBarExtra`,
  writing into `~/.lantern/config.toml`. V1.2.1.

### B-11-06 — `get.lantern.reiers.io` redirect to `install.sh`

- **Severity:** low — but required for the spec's one-liner UX.
- **What's missing:** `https://get.lantern.reiers.io` doesn't yet
  serve `install.sh`. The README and INSTALLER-SPEC reference this
  URL.
- **Plan:** Cloudflare worker or static site that serves
  `install.sh` from the current main branch (or from a tagged
  release once available). Trivial to set up; Nicklas action.

### Carry-overs from Phase 10 still open

- B-9-11 — Wallet keystore refuse-to-start when crypto disabled
- B-9-12 — F3 anchor freshness enforcement at daemon start (Phase 11
  partially addresses this via `bootstrap-anchor.json` recency; full
  enforcement still pending)
- B-9-14 — Deep block validation in net/blockpub
- B-9-15 — Header store ↔ F3 cert cross-check
- B-9-17 — Curio's `IPNI` task connectivity
- B-9-18 — `dev/dri/renderD128` permission spam
- B-10-01 — DHT auto-discovery for preferred Bitswap peers
- B-10-03 — At least one healthy public Lantern beacon online
- B-10-06 — Header sync via Bitswap (today still uses Glif)

None of these block V1.2 GA.
